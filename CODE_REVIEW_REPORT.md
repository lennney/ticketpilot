### ⚠️ 问题（需要改进）

#### **P0 - 严重问题**

**1. SQL注入风险**
```python
# src/ticketpilot/retrieval/vector_search.py
cur.execute(f"SET hnsw.ef_search = {ef_search}")
```
**风险：** 虽然`ef_search`是内部参数，但使用f-string拼接SQL是不安全的实践。  
**建议：** 使用参数化查询或验证输入范围。

**2. 缺少日志系统**
- 整个项目没有使用logging模块
- 无法追踪生产环境问题
- 调试困难

**建议：** 添加结构化日志，至少在关键路径（检索、生成、风险评估）添加日志。

---

#### **P1 - 重要问题**

**3. 函数过长**
以下函数超过150行，需要拆分：

| 函数 | 行数 | 建议 |
|------|------|------|
| `classifier.py` | 173行 | 拆分为多个子函数 |
| `prompt_builder.py` | 172行 | 提取模板逻辑 |
| `claim_guard.py` | 170行 | 分离检查规则 |
| `retrieval_metrics.py` | 163行 | 拆分计算逻辑 |
| `console.py` | 160行 | 分离UI和业务逻辑 |

**4. 缺少缓存机制**
- 只有10处缓存使用
- 重复查询没有缓存
- 影响性能

**建议：** 
- 添加查询结果缓存
- 使用`@lru_cache`装饰器
- 考虑Redis缓存层

**5. 异常处理不够细化**
- 只有11处异常捕获
- 缺少自定义异常类
- 错误信息不够友好

**建议：**
- 定义业务异常层次结构
- 添加异常处理器中间件
- 提供用户友好的错误信息

---

#### **P2 - 改进建议**

**6. 前端技术栈**
- 当前使用Streamlit（MVP）
- 交互体验有限
- 不够产品化

**建议：** 升级到React + FastAPI（参考InsightFlow）

**7. 缺少CI/CD配置**
- 没有.github目录
- 没有自动化测试流水线
- 没有自动部署

**建议：** 添加GitHub Actions配置

**8. 配置管理可以优化**
- 环境变量分散在多个文件
- 缺少配置验证
- 缺少配置文档

**建议：** 
- 统一配置管理（使用pydantic-settings）
- 添加配置验证
- 生成配置文档

## 三、安全性审查

2. **证据化生成** - 禁止幻觉，必须有引用
3. **ClaimGuard** - 五重安全检查
4. **密钥管理** - .env.local gitignore，.env.example占位符

2. **缺少输入验证** - 部分接口缺少参数验证
3. **缺少速率限制** - API接口无限制
4. **缺少CORS配置** - 前后端分离时需要
