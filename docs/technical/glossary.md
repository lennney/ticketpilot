# Glossary

## Terms

### RawTicket
The unprocessed input to the TicketPilot pipeline. Contains `original_text` (Chinese customer message), `submitted_at` timestamp, and optional `customer_id`. This is the sole entry point for all pipeline workflows.

### TicketOutput
The default output of the 4-stage pipeline (intake -> classification -> risk assessment -> retrieval). Contains all processing results including normalized text, intent classification, risk assessment, evidence candidates, and retrieval trace. Also the input to the optional draft generation stage.

### EvidenceCandidate
A Pydantic model representing a single knowledge chunk retrieved from the database as relevant evidence. Contains chunk metadata (chunk_id, doc_id, doc_type, source_id, source_table), content excerpt, RRF fusion score, rank position, and optional title. The boundary type between retrieval and pipeline consumers, hiding retrieval-internal ranking details.

### RetrievalTrace
An in-memory Pydantic model capturing the full retrieval trace: the query used, keyword/vector result counts, all fused results with per-ranker contribution breakdowns, and retrieval timestamp. Currently in-memory only — not persisted to the database.

### RRF (Reciprocal Rank Fusion)
A score-agnostic fusion algorithm that combines multiple ranked lists by summing reciprocal ranks. Formula: `score(doc) = sum(1 / (k + rank(ranker, doc)))`. Used in TicketPilot to combine keyword search results (PostgreSQL FTS) and vector search results (pgvector HNSW) into a single ranked list. See `src/ticketpilot/retrieval/rrf.py`.

### Fake Embedding
Deterministic pseudo-random vectors generated by `FakeEmbeddingProvider` using SHA-256 hashes of text as seeds for a random number generator. Produces 384-dimensional vectors in range [-1, 1]. These embeddings have **no semantic meaning** and are used for **pipeline verification only** — they prove the retrieval pipeline mechanics work (embedding generation, HNSW indexing, cosine similarity scoring) but provide zero semantic retrieval quality.

### pgvector
A PostgreSQL extension that adds vector storage and similarity search capabilities. Provides HNSW and IVFFlat index types with L2, cosine, and inner product distance operators. Used in TicketPilot for vector retrieval with cosine distance (`<=>` operator). Configured with HNSW index (m=16, ef_construction=200, ef_search=100).

### HNSW (Hierarchical Navigable Small World)
A graph-based approximate nearest neighbor (ANN) index algorithm for vector similarity search. Provides fast search with configurable recall/speed tradeoffs. Key parameters: `m` (max connections per node), `ef_construction` (index build quality), `ef_search` (search breadth). Used in TicketPilot via pgvector for vector retrieval.

### DraftReply
A Pydantic model representing a generated draft reply. Contains the drafted text, associated citations with evidence references, unsupported claims flagged by the CitationValidator, missing information notes, confidence score, and safety flags (`must_human_review`, `fallback_reason`). Output of the optional draft generation stage.

### CitationValidator
A deterministic regex-based guardrail that validates citation correctness and detects potentially unsupported claims in draft replies. Performs three checks: citation marker existence (every `[N]` has a matching `Citation`), claim-coverage scan (sentences with Chinese claim keywords without citation markers), and cross-reference against evidence candidates. Not a full NLP claim verifier.

### ReviewDecision
A Pydantic model representing a self-contained audit record of a human review action. Captures a snapshot of the ticket, draft, and decision state at review time, including the review action (APPROVE/EDIT/ESCALATE/REJECT), original draft text, reviewer edits, decision reason, risk flags, confidence, and timestamps. Persisted to append-only JSONL via ReviewStore.

### ReviewStore
An append-only JSONL persistence layer for ReviewDecision records. Provides three operations: `save()` (append one record), `load_all()` (read all records), `count()` (record count). Designed for local MVP use; the interface supports replacement with a database-backed implementation for production.

### Human-in-the-loop
A design pattern where automated system outputs require human review before any action is taken. In TicketPilot, this is enforced by: (a) `must_human_review=True` for any ticket with risk flags, unsupported claims, or insufficient evidence; (b) the review console has no send functionality; (c) all review decisions are recorded to an append-only audit trail. This is a safety constraint, not a missing feature.

### Quality Gate
A bash script (`scripts/run_quality_gate.sh`) that enforces code quality standards before acceptance. Runs: Ruff linting, unit tests (with >= 70% coverage), integration tests (with skip-count guard: 0 skipped = pass), OpenSpec validation, and secret scanning. Any failure exits non-zero. The gate was hardened during audit remediation (Stage 04) — previously every check used `|| true` making it a no-op.

### OpenSpec Change
A structured, traceable change management artifact used throughout TicketPilot development. Each change follows: proposal -> design -> spec -> tasks -> implementation -> quality gate -> archive. Each change has its own directory under `openspec/changes/` with a design document, spec directory, and task breakdown. Archived changes are moved to `openspec/changes/archive/`. The pattern prevents uncontrolled code generation and requirement drift.
