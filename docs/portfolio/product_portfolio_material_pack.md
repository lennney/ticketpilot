## 1. 项目一句话定位

### 简历版

> 面向中文电商客服场景的工单分诊与证据化草稿辅助系统——基于意图分类、风险识别、分层知识检索、人工审核和离线评测构建 AI 产品原型。经历真实嵌入升级、评估驱动知识优化、检索排名诊断、LLM 草稿生成等迭代阶段。

### 作品集版

> TicketPilot 是一个中文客服工单分诊与证据化回复 Copilot 原型。它把非结构化的客服消息转化为结构化工单，依次完成意图分类（8 类）、风险识别（8 种标记 + 三级严重度）、分层知识检索（FAQ / Policy / Case）、证据驱动草稿生成，最后通过人工审核决策闭环。项目经历 5 个主要迭代阶段：MVP 数据与评测体系（Phase 7）→ 真实嵌入升级与检索对比（Phase 8）→ 评估驱动知识优化与 Provider Identity Gate（Phase 9）→ 检索排名诊断与细粒度评测（Phase 10）→ 证据约束 LLM 草稿生成（Phase 11，已完成）。101 条评测工单 + 106 条知识记录构成完整的数据与评测基底。

### 面试口述版

> "TicketPilot 是我做的一个中文客服工单处理系统原型。用户提交一条中文售后消息，系统会先分类——这是退款、换货还是投诉。然后判断风险——有没有法律威胁、赔偿要求、隐私泄露。接着去知识库里检索相关的政策条款和相似案例，基于这些证据生成一个草稿回复。但草稿不会自动发送——高风险工单强制人工审核。整个系统默认不调用 LLM，不连接发送通道，所有评测都有离线数据支撑。
>
> 到目前为止项目经历了 5 个迭代阶段：从建立 101 条数据基底、到接入真实中文嵌入对比检索效果、到定向补知识并发现一个配置静默回退的 bug、到精细化评测发现 78% 的"错误"其实是评测粒度问题、再到正在做的证据约束 LLM 草稿生成。每个阶段都在回答一个具体的产品问题——不是简单堆功能。"

## 2. 产品背景与问题定义

### 用户痛点

电商客服团队每天处理大量重复性售后工单：退款进度查询、退换货申请、物流催单、发票开具。这些工单看似简单，但每个都可能隐藏投诉升级、法律威胁或隐私泄露风险。一线客服需要在短时间内判断工单类型、评估风险、查找相关政策、撰写回复——高强度、高重复、高压力。

### 业务痛点

- **分诊不一致**：同一个"退款"问题，不同客服可能给出不同处理路径
- **风险遗漏**：一条普通退款消息里带着"不然我找律师"，如果没识别出法律风险，可能演变成严重投诉
- **回复质量不统一**：靠个人经验写回复，新客服培训成本高
- **无追溯**：工单处理完就结束了，没有结构化的决策记录可供复盘

### 风险痛点

客服场景中高风险信号隐蔽：用户说"我要赔偿"不只是抱怨，说"我账号被异地登录"不只是技术问题。如果系统不能识别这些信号、直接自动回复、或者客服遗漏处理，后果可能从一次投诉升级到法律纠纷。

### AI 落地痛点

- 纯 FAQ 机器人只能回答标准问题，无法处理复合意图（退款 + 投诉 + 赔偿）
- 通用 RAG 问答能检索文档，但没有风险判断和流程控制
- 全自动客服机器人风险高——一旦自动发送了错误承诺或无视了法律威胁，后果严重
- 大多数 AI demo 只有"输入 → 输出"，缺乏评测体系、人工审核闭环和边界约束

### 所以 TicketPilot 选择

> 不做聊天机器人，不做全自动客服。
> 做一个"AI 辅助分诊 + 风险门控 + 证据驱动草稿 + 人工审核确认"的产品原型。
> 用评测数据证明它不只是能跑，而是有明确边界、可追溯、可评测。

## 3. 目标用户与使用场景

### 目标用户画像

- **中小型电商平台**：日均 100–1000 条售后工单，需要工单分诊和辅助回复
- **售后客服团队**：10–50 人团队，需要统一处理标准和风险管控
- **客服主管 / 质检人员**：需要审核高风险工单、回溯处理记录
- **AI 产品 / 研发团队**：需要评估 AI 辅助效果、迭代模型和策略

### 使用者角色

| 角色 | 职责 | 使用 TicketPilot 的方式 |
|------|------|------------------------|
| 一线客服 | 处理日常售后工单 | 接收系统建议的草稿，确认或修改后提交 |
| 客服主管 | 审核高风险工单 | 在人工审核控制台上审批评级、赔偿、隐私等场景 |
| 质检/运营 | 评估团队处理质量 | 查看评估报告、分析风险标记覆盖率、回溯决策记录 |
| AI 产品团队 | 优化分类和检索策略 | 运行离线评估、调整规则、对比指标变化 |

### 典型使用场景

| 场景 | 示例工单内容 | 系统行为 |
|------|-------------|----------|
| 售后退款 | "我买的手机有质量问题，申请退款" | 分类→refund，低风险，证据化草稿 |
| 投诉升级 | "再不处理我就找律师起诉" | 分类→complaint，legal_risk，强制人审 |
| 隐私/账号异常 | "我的账号在异地登录，个人信息可能泄露了" | 分类→account_issue，privacy+security风险，强制人审 |
| 发票/支付争议 | "付款成功但订单未确认，发票开不出来" | 分类→refund/other，policy_conflict，有风险信号时人审 |
| 证据不足转人工 | "退款。"（仅两字） | 分类→refund，insufficient_evidence，触发 fallback |

## 4. 产品目标与非目标

### Product Goals

- 提高工单分诊一致性——确保同一类型的工单获得统一的分类和处理路径
- 识别高风险工单——从自然语言中检测投诉、赔偿、法律、隐私等风险信号
- 帮助客服基于证据生成草稿——回复引用政策条款和相似案例，减少空口承诺
- 保留人工审核为最终决策层——高风险场景强制路由到人，不自动发送
- 建立可评估的数据闭环——101 条评测工单 + 离线评估流水线，可量化对比每次改动
- 通过迭代验证产品判断——不是盲目换模型/堆数据，而是通过评测分析确定下一步瓶颈

### Non-goals（明确不做）

- ❌ 不做全自动客服——所有回复都是草稿，不自动发送
- ❌ 不替代人工客服——系统是辅助决策，不是替代人
- ❌ 不做生产系统——当前是本地 Demo / 作品集原型
- ❌ 不接真实企业数据——所有工单和知识记录都是合成/改编的
- ❌ 不声称真实线上效果——评测基于本地确定性数据，非线上 A/B test
- ❌ 默认不用真实嵌入验证语义检索——fake embedding 仅验证管道机械正确性；真实嵌入通过 opt-in 提供
- ❌ 不接真实 LLM——草稿用模板生成，LLM provider 接口仅用于管道验证

## 5. 核心产品流程

### 流程图（文本版）

```
用户发送客服消息
     │
     ▼
[工单接收与标准化]
  → 文本规范化
  → 提取订单号、客户标识
     │
     ▼
[意图分类] — 8 类
  → 退款 / 退换货 / 账号 / 投诉 / 物流 / 技术 / 咨询 / 其他
     │
     ▼
[风险评估] — 8 种标记 + 3 级严重度
  → complaint / compensation / legal / privacy
  → account_security / policy_conflict / insufficient_evidence / low_confidence
  → severity: LOW → MEDIUM → HIGH
     │
     ▼
[分层知识检索]
  → 关键词全文搜索 + 向量相似度 + RRF 融合排序
  → 检索范围: FAQ(41条) + Policy(34条) + Case(31条)
     │
     ▼
[证据化草稿生成]
  → 基于检索到的 Policy/Case 生成回复草稿
  → 引用具体证据来源
  → ClaimGuard: 声明校验 + 禁止承诺检测 + 风险感知检查
     │
     ▼
[人工审核]
  → 有风险标记 → must_human_review = true
  → 审核操作: 批准(Approve) / 编辑(Edit) / 升级(Escalate) / 拒绝(Reject)
  → 每次审核写入 ReviewDecision JSONL（审计追踪）
     │
     ▼
[离线评测与复盘]
  → CSV 模式: 加载样本预测 → 计算 7 项指标
  → Pipeline 模式: 运行完整管道 → 计算指标
  → 检索对比模式: Fake vs Real embedding 对比
  → 输出 JSON + Markdown 报告
```

### 每一步的产品价值

| 流程步骤 | 产品价值 | 对应系统能力 |
|----------|---------|-------------|
| 工单接收与标准化 | 让非结构化消息变成可处理的结构化数据 | intake/normalizer, entity_extractor |
| 意图分类 | 自动判断工单类型，减少人工分类成本 | classification/ 模块 |
| 风险评估 | 发现隐蔽风险信号，防止风险遗漏 | risk/assessor + rules |
| 分层知识检索 | 基于政策而非经验生成回复，提高一致性 | retrieval/ 完整链路 |
| 证据化草稿生成 | 客服不用从零写回复，提高效率和质量 | drafting/ provider + validator + guard |
| ClaimGuard | 防止系统编造不存在的政策或做出禁止承诺 | drafting/claim_guard |
| 人工审核 | 高风险场景不自动处理，保留人的最终判断 | review/ console + store |
| 离线评测与复盘 | 每次改动可量化对比，不是"感觉变好了" | evaluation/ 全模块 |

## 6. 功能模块拆解

### 模块 1：工单输入与标准化

| 维度 | 说明 |
|------|------|
| 用户价值 | 直接粘贴客服消息即可处理，无需手动整理格式 |
| 业务价值 | 统一入口，保证后续流程的输入一致性 |
| 技术支撑 | RawTicket → NormalizedTicket 转换，实体提取（订单号、客户 ID） |
| 当前边界 | 仅支持单条文本输入，不支持批量导入和附件 |

### 模块 2：意图分类

| 维度 | 说明 |
|------|------|
| 用户价值 | 系统自动告诉客服"这是退款工单"还是"这是投诉工单" |
| 业务价值 | 减少人工分类时间，统一分诊标准 |
| 技术支撑 | 基于规则的 8 类分类器，不使用 LLM，确定性输出 |
| 当前边界 | 不支持自定义分类；单意图分类（复合意图归为主类） |

### 模块 3：风险识别与人审门控

| 维度 | 说明 |
|------|------|
| 用户价值 | 系统自动标记高风险工单，不需要客服逐条排查 |
| 业务价值 | 降低风险遗漏概率，保护平台免受法律/合规风险 |
| 技术支撑 | 8 种风险标记规则 + 严重度自动升级 + must_human_review 门控 |
| 当前边界 | 基于关键词规则，不是 ML 模型；部分边界案例可能漏报或误报 |

### 模块 4：FAQ / Policy / Case 分层知识库

| 维度 | 说明 |
|------|------|
| 用户价值 | 客服在一个系统里查标准问答、政策条款和历史案例，不用来回切换 |
| 业务价值 | 知识集中管理，回复基于最新政策而非个人经验 |
| 技术支撑 | 三种独立 doc_type，各带业务域标签，统一 chunk 存储 |
| 当前边界 | 106 条种子数据（FAQ=41, Policy=34, Case=31），不覆盖全部业务场景；非真实企业数据 |

### 模块 5：Hybrid Retrieval + RRF 证据检索

| 维度 | 说明 |
|------|------|
| 用户价值 | 系统自动找到最相关的政策条款和相似案例，减少客服查找时间 |
| 业务价值 | 回复一致性提升，证据可追溯 |
| 技术支撑 | 关键词全文搜索 + pgvector HNSW 向量检索 + RRF 融合排序 |
| 当前边界 | 默认使用 fake embedding，可通过环境变量切换至 DashScope text-embedding-v4 等真实嵌入 |

### 模块 6：Evidence-based Draft

| 维度 | 说明 |
|------|------|
| 用户价值 | 客服看到草稿时知道"系统为什么这么写"——有证据引用 |
| 业务价值 | 减少空口承诺风险，新客服可参考草稿学习正确回复 |
| 技术支撑 | FakeDraftProvider / FakeLLMProvider + PromptBuilder + CitationValidator |
| 当前边界 | 默认模板生成；LLM provider 接口已定义，FakeLLMProvider 用于离线验证 |

### 模块 7：ClaimGuard（声明校验）

| 维度 | 说明 |
|------|------|
| 用户价值 | 草稿不会编造政策或做出禁止承诺——客服可以信任内容是有依据的 |
| 业务价值 | 降低法律风险，防止系统做出平台无法兑现的承诺 |
| 技术支撑 | CitationValidator + ClaimGuard（声明校验、禁止承诺检测、风险感知检查） |
| 当前边界 | 基于规则扫描，复杂 claim 可能漏检 |

### 模块 8：Streamlit 人审控制台

| 维度 | 说明 |
|------|------|
| 用户价值 | 审核员在一个界面看到工单详情、风险评估、证据和草稿，直接操作 |
| 业务价值 | 完整的审核工作流 + 决策审计追踪（JSONL） |
| 技术支撑 | Streamlit 界面 + ReviewDecision 持久化 + JSONL 存储 |
| 当前边界 | MVP 级别，无登录、无多用户、无自动刷新 |

### 模块 9：Evaluation Pipeline

| 维度 | 说明 |
|------|------|
| 用户价值 | 产品团队可量化对比每次改动好不好 |
| 业务价值 | 不是"感觉变好了"而是数字说话——intent accuracy、risk flag recall、Doc-ID Recall 等 |
| 技术支撑 | CSV 模式 + Pipeline 模式 + 检索对比模式，7 项指标 + 细粒度 doc-ID 指标 |
| 当前边界 | 101 条数据集不足以做统计显著分析；指标受限于规则组件而非 ML |

### 模块 10：Demo Scenario Pack

| 维度 | 说明 |
|------|------|
| 用户价值 | 面试/作品集展示时，有 3 个完整场景可讲，不是零散截图 |
| 业务价值 | 向 stakeholders 展示产品能力时有一条清晰的叙事线 |
| 技术支撑 | 3 个场景文档 + 示例工单 + 预期行为描述 + 限制说明 |

## 7. 数据与评测体系说明

### 数据规模

| 数据集 | 数量 | 说明 |
|--------|:----:|------|
| Eval tickets（评测工单） | 101 | 合成中文单轮客服工单 |
| Golden expectations（黄金标注） | 101 | 产品规则驱动的预期行为标注 |
| Sample predictions（样本预测） | 101 | 全标记 no_auto_send=true |
| Knowledge records（知识记录） | 106 | FAQ=41 / Policy=34 / Case=31 |
| 覆盖的意图类型 | 8 | refund, return_exchange, account_issue 等 |
| 覆盖的风险标记类型 | 8 | complaint, compensation, legal, privacy 等 |
| 覆盖的严重度级别 | 3 | LOW=53, MEDIUM=33, HIGH=15 |
| 多意图工单 | 7 | refund+complaint, account+privacy 等 |
| 边界情况工单 | 5 | 单字符、超长文本、特殊字符等 |

### 数据来源策略

数据构造遵循"公开参考 → AI 辅助提取 → 人工改写 → 人工标注"四级流程：

```
公开/开源客服数据集（CSDS / Kaggle / 中文对话语料 / 公开政策页面）
    │ （作为场景和表达参考，不提交原始数据）
    ▼
AI 辅助字段提取（提取场景、风险信号、意图候选）
    │ （AI 是提取辅助，不是最终标注者）
    ▼
人工改写为合成工单（单轮中文客服工单，无真实个人信息）
    │ （人工改写，非原始数据复制）
    ▼
人工标注 Golden Expectations（基于产品规则，非 AI 生成）
    │
    ▼
进入评估流水线
```

关键原则：
- **不提交原始外部数据**到仓库
- **不把 AI 输出当最终标签**——所有 golden 标注经人工确认
- **不使用真实个人信息**——工单中均为占位数据

### 评测维度

| 维度 | Pipeline 模式结果 | 说明 |
|------|:----------------:|------|
| Intent accuracy | 53.5% | 意图分类是否正确 |
| Severity accuracy | 54.5% | 严重度级别是否正确 |
| Must-human-review accuracy | 53.5% | 是否需要人工审核判断是否正确 |
| Risk flag precision | 33.3% | 风险标记精确率 |
| Risk flag recall | 27.0% | 风险标记召回率 |
| Risk flag F1 | 29.8% | 风险标记 F1 |
| Evidence doc type recall | 43.2% | 证据文档类型召回率 |
| Fallback correctness | 90.1% | 降级处理是否正确 |
| No-auto-send compliance | **100.0%** | 无自动发送合规率 |

### 评测结果解读（产品视角）

**no-auto-send compliance = 100%** 是架构约束，不是回复质量指标。系统不连接任何发送通道，所以这条必然是 100%。它证明了"架构对齐"而非"效果很好"。

**Intent / Severity 指标（~53–54%）** 反映了规则组件的确定性行为。这些指标说明"评测体系已建立"，但不能被包装成生产级效果指标。

**Risk flag 指标（F1 ~30%）** 反映了关键词规则对边界场景的覆盖不足。

**CSV 模式全部 100%** 说明当预测与 golden 严格对齐时，指标计算正确——验证了评测工具链本身无 bug。

### Phase 8–10 关键评测发现

**Phase 8：真实嵌入对比**
- Top-1 hit rate: 31.7%（Fake 384-d）→ 42.6%（Real 1024-d），提升 +10.9%
- MRR: 0.4114 → 0.4913，提升 +0.0799
- Top-10 不变（同为 59.4%）——更好的 embedding 也无法补全知识库本身的缺口
- Real embedding 改善 ranking，但 41 个 wrong cases 完全一致、均为 missing_doc_type
- **产品判断：当前主要瓶颈是知识覆盖，不是 embedding 质量**

**Phase 9：评估驱动知识优化**
- 41 个 wrong cases → 8 类故障分类 → 24 个知识缺口
- 定向补充 11 条 P0 知识记录，P0 hit rate 达 75.0%
- 关键发现：`load_dotenv()` 从未被调用，导致所有评测静默回退 fake provider
- 修复后真实 embedding 评测结果翻转，Top-1 提升 2.0%
- **产品判断：Provider Identity Gate 必须建立——不能相信你不知道来源的指标**

**Phase 10：检索排名诊断与细粒度评测**
- Doc-ID golden labels 从 14 扩展到 86 个评测用例
- Doc-ID Recall@10 = 91.9%，较 doc-type 指标提升 32.5 个百分点
- 32/41 个 doc-type wrong cases 被重新归类为 doc_id-found（78%）
- **核心结论：78% 的"错误"案例实为评测粒度问题，而非检索系统失效**

### 迭代逻辑总结

```
Phase 7: 建数据基底和评测体系
  → Phase 8: 换真实嵌入，发现瓶颈在知识覆盖
    → Phase 9: 补知识，发现配置静默回退问题
      → Phase 10: 精细化评测维度，发现大部分"错误"是粒度问题
        → Phase 11: 完成证据约束草稿生成，构建 8 层安全架构（已完成）
```

## 9. 三个强 Demo 场景包装

### Demo 1：退款投诉

#### 作品集说明版

> 一条常见的售后退款消息，如果包含"投诉""赔偿""找律师"等语言，就从一个普通工单变成了高风险工单。TicketPilot 能同时检测三类风险标记——投诉风险、赔偿风险、法律风险——自动将严重度升级为 HIGH，强制路由到人工审核。草稿基于检索到的赔偿政策和相似案例生成，不凭空承诺赔偿金额。这个场景展示了系统的多意图识别能力、复合风险检测能力和高风险门控机制。

#### 面试口述版

> "退款投诉场景是我的首选 demo。用户说'退款三天没到账，不解决就投诉找律师'——一条消息里同时带着退款诉求、投诉情绪和法律威胁。系统会把它分类为 refund/complaint，识别出投诉、赔偿、法律三个风险标记，严重度自动升到 HIGH。这个时候系统不会自动回复，而是强制交给人工审核。草稿引用的是知识库里关于投诉升级和律师函处理的政策条款，不是编的。这个场景能很清楚地展示多风险叠加的判断逻辑。"

### Demo 2：隐私/账号异常

#### 作品集说明版

> "我的账号被异地登录了，手机号和个人信息可能泄露了"——隐私安全类工单是最敏感的场景。TicketPilot 能从自然语言中检测账号安全风险和隐私泄露风险两个标记，判定 HIGH severity 并强制人工审核。草稿基于隐私政策和安全流程生成，提供账号保护指引但不代用户执行敏感操作（如改密码、冻结账号）。这个场景体现了系统在数据安全和个人信息保护上的边界意识。

#### 面试口述版

> "隐私场景是我最展示系统边界意识的场景。用户报告异地登录和信息泄露，系统要识别出两个风险标记——account_security_risk 和 privacy_risk，判 HIGH，强制人审。草稿会引用隐私泄露处理规则和账号安全案例，但注意——系统不会在草稿里说'已经帮你改了密码'，因为它没有权限做这些操作，也不会承诺赔偿。它的边界很清晰：只提供信息和建议，具体操作交给人工。这让面试官能看到我设计产品时考虑过 AI 的权限边界。"

### Demo 3：发票/支付争议

#### 作品集说明版

> "付款成功但订单未确认、重复扣款、发票开不出来"——支付和发票问题不需要每次都由人工处理。TicketPilot 能区分场景：纯发票咨询无风险标记时系统自动处理；包含投诉语言或政策冲突时触发人工审核。草稿基于发票政策和支付政策生成，与平台规则保持一致，不主动承诺退款或赔偿。这个场景展示了条件性人工审核逻辑——不是所有工单都走人审，系统能判断何时自动处理、何时升级。

#### 面试口述版

> "发票/支付场景是我展示条件性审核逻辑的例子。不是所有工单都需要人工处理——纯发票咨询系统可以自动回复。但当工单出现 policy_conflict 或 complaint_risk 时，系统会自动触发人审。这个设计是有意为之的——让简单的工单自动流转降低客服负担，让复杂或有风险的工单人工处理保证质量。草稿只基于检索到的发票政策生成，不会说'放心，肯定能退'这种话。"

## 10. 产品差异化

### 对比表

| 维度 | 普通 FAQ Bot | 普通 RAG 问答 | 全自动客服机器人 | TicketPilot |
|------|:-----------:|:------------:|:--------------:|:----------:|
| 工单分诊 | ❌ | ❌ | ✅ | ✅ |
| 意图分类 | ❌ | ❌ | 部分 | **8 类确定性分类** |
| 风险识别 | ❌ | ❌ | 有限 | **8 种风险标记** |
| 知识分层 | ❌ | 单一文档库 | 混合 | **FAQ / Policy / Case** |
| 证据引用 | ❌ | 有（不可控） | 部分 | **显式引用 + 校验 + 禁止承诺检测** |
| 人工审核 | ❌ | ❌ | 仅投诉升级 | **内置 + 必须** |
| 离线评测 | ❌ | ❌ | ❌ | **7 项指标 + 细粒度检索评测** |
| 迭代验证 | ❌ | ❌ | ❌ | **5 阶段迭代，每阶段有明确产品判断** |
| 边界约束 | ❌ | ❌ | ❌ | **no-auto-send 架构级** |
| 适合作品集 | 一般 | 常见 | 一般 | **适合（展示 AI 产品设计 + 迭代思维）** |

### 结论

TicketPilot 的优势不是"更强的聊天能力"，而是：

> **把 AI 放进客服工单处理流程，并用风险标记、人工审核、证据引用、离线评测和迭代验证来约束它。**

它不是通用聊天机器人，不是文档 QA，不是全自动客服。它是一个**带边界、带流程、带评测、带迭代轨迹的 AI 辅助决策原型**。这个定位让它特别适合作品集展示——因为它展示的不是"AI 能回答问题"，而是"AI 产品经理如何设计一个可控、可评测、可迭代的 AI 工作流"。

## 13. 面试讲述

### 1 分钟版本

> "TicketPilot 是我做的一个中文客服工单处理系统原型。用户发一条售后消息，系统会先分类——这是退款、换货还是投诉——然后判断风险，有没有法律威胁、赔偿要求之类的。接着去知识库里查相关政策，基于证据生成一个草稿回复。但草稿不会自动发送，高风险工单强制人工审核。整个过程默认不调用 LLM，不接真实企业数据。
>
> 这个项目最值得讲的是它的迭代逻辑。不是一次性建完的，而是 5 个阶段：先建 101 条数据和评测体系，再换真实嵌入发现瓶颈在知识覆盖，然后定向补知识时发现了一个配置静默回退的 bug，接着精细化评测发现 78% 的错误其实是评测粒度问题，现在基于足够好的检索质量在加 LLM 草稿生成。每个阶段回答一个具体的产品问题。"

### 3 分钟版本

> "TicketPilot 的出发点很明确：大多数 AI demo 都在展示'AI 能干什么'，我想展示的是'AI 产品的边界该怎么设计，以及怎么通过迭代验证做判断'。
>
> 产品流程很简单：用户提交客服消息→系统标准化→意图分类（8 类）→风险判断（8 种标记）→知识检索（FAQ/Policy/Case 三层）→生成带证据引用的草稿→ClaimGuard 校验→人工审核决策。不做全自动客服，不自动发送。
>
> 但比流程更重要的是迭代过程。项目分 5 个阶段推进。Phase 7 打数据基底——从 10 条工单扩展到 101 条，建了评测流水线。Phase 8 换真实嵌入——DashScope text-embedding-v4 对比 fake，Top-1 从 31.7% 到 42.6%。但更有意思的是错例分析发现：fake 和 real 的 41 个错例完全一致，都是 missing_doc_type。这说明瓶颈不在模型，在知识覆盖。
>
> 于是 Phase 9 去补知识——做了 8 类故障分类、24 个知识缺口映射、加了 11 条 P0 记录。但评测结果反而差了。查了半天发现 `load_dotenv()` 没被调用——.env.local 的配置根本没生效，所有评测静默回退到 fake。这个 bug 让我建了 Provider Identity Gate——运行时验证实际 provider 身份。
>
> Phase 10 发现另一个问题：doc-type 级别的评测太粗糙。然后建了 doc-ID 级别的细粒度标注，结果 Doc-ID Recall@10 达到 91.9%。78% 的所谓"错误"其实是评测粒度问题——文档类型对了，只是具体文档 ID 和 golden 标的不是同一个。
>
> 现在 Phase 11 在做证据约束的 LLM 草稿生成。因为检索质量已经够用了，下一步是让草稿更像真人客服写的，同时用 ClaimGuard 确保不编造政策。
>
> 这个迭代故事比任何单点指标都重要——它展示了怎么用评测做产品判断，而不是盲目堆技术。"

### 追问回答

**你为什么做这个项目？**
> "初衷不是'我要做个客服 AI'，而是我想做一个能展示 AI 产品设计能力的项目——一个可控、可评测、有边界约束、有迭代逻辑的 AI 工作流。客服工单场景刚好合适：流程清晰、风险明确、有业务判断逻辑。"

**这个项目解决了什么问题？**
> "它解决的是'AI 辅助决策怎么设计边界和迭代路径'的问题——不是 AI 能不能回答问题，而是 AI 怎么判断什么时候该回答、什么时候该转人工、回答的依据来自哪里、以及怎么通过评测判断下一步该做什么。"

**数据从哪里来？**
> "全部是合成数据。我们从 CSDS、Kaggle 等公开数据集取场景和表达参考，但不提交原始数据。每条工单都是人工改写的，不使用真实个人信息。Golden expectations 也是人工根据产品规则标注的。AI 只做字段提取辅助，不是最终标注者。"

**怎么评估？**
> "7 项指标 + 细粒度 doc-ID 评测。CSV 模式和 Pipeline 模式双通道。额外有检索对比模式支持 Fake vs Real embedding 对比。no-auto-send=100% 是架构约束。Doc-ID Recall@10=91.9% 是 Phase 10 细粒度评测的核心指标。"

**Phase 8 学到什么？**
> "最大的收获不是 Top-1 提升了 10%，而是发现 fake 和 real embedding 的 41 个 wrong cases 完全一致。这说明当时系统的瓶颈不在 embedding 质量，而在知识库本身有没有覆盖这些场景。这个判断比单纯报一个更高的准确率更有产品价值。"

**Phase 9 的 Provider Identity Gate 是什么？**
> "一个运行时检查机制——每次跑评测时记录实际使用的 provider 身份。因为在 Phase 9 中发现 python-dotenv 装好了但没调用，所有环境变量配置都没生效，系统静默回退到了 fake provider。这导致我们基于错误的数据做了好几轮迭代决策。教训是：你不能相信你不知道来源的指标。"

**Phase 10 证明了什么？**
> "证明了 doc-type 级别的评测太粗糙。78% 的 wrong cases 其实是评测粒度问题——系统其实找到了正确的文档类型，但 golden 标的 doc ID 和系统找到的不一样。换成 doc-ID 粒度后 Recall@10 从 59.4% 跳到 91.9%。这告诉我们：在说'系统不好'之前，先确认你的评估方法对不对。"

**为什么不做全自动客服？**
> "因为全自动客服的技术和业务风险都很高：错误回复、法律承诺、无人工兜底。TicketPilot 的定位就是 AI 辅助+人审兜底。'不自动发送'是架构级的约束，不是配置开关。"

**当前效果有什么限制？**
> "主要限制是规则组件（分类器和风险规则）的覆盖有限，intent/severity 指标 ~53% 反映了这个现实。另外 101 条工单规模还不够做统计分析。数据全是合成的，不代表真实业务分布。"

**下一步怎么优化？**
> "Phase 11 收尾——完成 LLM 草稿生成和离线草稿评估。然后回到 Phase 10 诊断发现的 7 个 zero-hit 案例做 query expansion，32 个 partial-hit 案例做 fusion ranking 优化。"

**这个项目体现了你的产品能力在哪里？**
> "第一，产品定位——有明确边界（不做全自动客服）和迭代路径（5 阶段）。第二，需求拆解——把'客服 AI'拆成了分类、风险、检索、草稿、人审、评测六个独立可迭代的模块。第三，数据策略——四级数据构造流程。第四，评测体系——多维指标 + 细粒度评测。第五，迭代验证——每次通过评测分析做下一步决策，不是堆功能。第六，基础设施意识——发现并修复了 Provider Identity Gate 这样的系统性问题。"

## 15. 项目边界提醒

### 不能这么说 ❌

| 不能说的话 | 原因 |
|-----------|------|
| "这是一个生产级客服系统" | 它是本地 Demo / 作品集，未经生产验证 |
| "已通过真实企业验证" | 所有数据是合成的，无真实企业数据 |
| "使用真实客户数据" | 使用的是合成数据，无真实 PII |
| "系统自动发送客服回复" | 不连接任何发送通道，no-auto-send 是架构约束 |
| "可以替代人工客服" | 系统是辅助决策，不能替代人的判断 |
| "已通过线上 A/B 测试验证效果" | 评测是离线确定性的，非在线验证 |
| "达到行业 benchmark 水平" | 无行业 benchmark 对比，小规模合成数据不具可比性 |
| "真实语义检索效果已完全验证" | 真实嵌入通过 opt-in 使用，且评估基于合成数据 |

### 应该这么说 ✅

| 正确的说法 | 原因 |
|-----------|------|
| "这是一个本地 Demo / 作品集原型" | 准确描述成熟度 |
| "使用合成/改编数据" | 说明数据来源策略 |
| "默认使用 fake embedding 验证管道机械正确性；真实嵌入可 opt-in" | 说明嵌入的定位和可切换性 |
| "草稿仅用于人工审核参考，不自动发送" | 说明 no-auto-send 是架构约束 |
| "AI 辅助客服决策，不替代人工" | 说明人机协作定位 |
| "离线评测验证产品流程正确性，非线上效果" | 说明评测定位 |
| "当前指标反映规则组件行为，不能包装为生产级效果" | 诚实说明限制 |
| "项目展示了如何通过评测分析做迭代决策而非盲目堆功能" | 描述项目的真实价值 |
