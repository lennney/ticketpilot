## 1. 项目一句话定位

### 简历版

> 面向中文电商客服场景的工单分诊与证据化草稿辅助系统——基于意图分类、风险识别、分层知识检索、人工审核和离线评测构建 AI 产品原型。

### 作品集版

> TicketPilot 是一个中文客服工单分诊与证据化回复 Copilot 原型。它把非结构化的客服消息转化为结构化工单，依次完成意图分类（8 类）、风险识别（8 种标记 + 三级严重度）、分层知识检索（FAQ / Policy / Case）、证据驱动草稿生成，最后通过人工审核决策闭环。全流程不使用 LLM 和真实嵌入服务。101 条评测工单 + 95 条知识记录 + 离线评估流水线构成完整的数据与评测基底。

### 面试口述版

> "TicketPilot 是我做的一个中文客服工单处理系统原型。用户提交一条中文售后消息，系统会先分类——这是退款、换货还是投诉。然后判断风险——有没有法律威胁、赔偿要求、隐私泄露。接着去知识库里检索相关的政策条款和相似案例，基于这些证据生成一个草稿回复。但草稿不会自动发送——高风险工单强制人工审核。整个过程不调用 LLM，不连接发送通道，所有评测都有离线数据支撑。这不是全自动客服，而是一个带风险判断、证据追溯和人工审核闭环的 AI 辅助决策流程。"

## 2. 产品背景与问题定义

### 用户痛点

电商客服团队每天处理大量重复性售后工单：退款进度查询、退换货申请、物流催单、发票开具。这些工单看似简单，但每个都可能隐藏投诉升级、法律威胁或隐私泄露风险。一线客服需要在短时间内判断工单类型、评估风险、查找相关政策、撰写回复——高强度、高重复、高压力。

### 业务痛点

- **分诊不一致**：同一个"退款"问题，不同客服可能给出不同处理路径
- **风险遗漏**：一条普通退款消息里带着"不然我找律师"，如果没识别出法律风险，可能演变成严重投诉
- **回复质量不统一**：靠个人经验写回复，新客服培训成本高
- **无追溯**：工单处理完就结束了，没有结构化的决策记录可供复盘

### 风险痛点

客服场景中高风险信号隐蔽：用户说"我要赔偿"不只是抱怨，说"我账号被异地登录"不只是技术问题。如果系统不能识别这些信号、直接自动回复、或者客服遗漏处理，后果可能从一次投诉升级到法律纠纷。

### AI 落地痛点

- 纯 FAQ 机器人只能回答标准问题，无法处理复合意图（退款 + 投诉 + 赔偿）
- 通用 RAG 问答能检索文档，但没有风险判断和流程控制
- 全自动客服机器人风险高——一旦自动发送了错误承诺或无视了法律威胁，后果严重
- 大多数 AI demo 只有"输入 → 输出"，缺乏评测体系、人工审核闭环和边界约束

### 所以 TicketPilot 选择

> 不做聊天机器人，不做全自动客服。
> 做一个"AI 辅助分诊 + 风险门控 + 证据驱动草稿 + 人工审核确认"的产品原型。
> 用评测数据证明它不只是能跑，而是有明确边界、可追溯、可评测。

## 3. 目标用户与使用场景

### 目标用户画像

- **中小型电商平台**：日均 100–1000 条售后工单，需要工单分诊和辅助回复
- **售后客服团队**：10–50 人团队，需要统一处理标准和风险管控
- **客服主管 / 质检人员**：需要审核高风险工单、回溯处理记录
- **AI 产品 / 研发团队**：需要评估 AI 辅助效果、迭代模型和策略

### 使用者角色

| 角色 | 职责 | 使用 TicketPilot 的方式 |
|------|------|------------------------|
| 一线客服 | 处理日常售后工单 | 接收系统建议的草稿，确认或修改后提交 |
| 客服主管 | 审核高风险工单 | 在人工审核控制台上审批评级、赔偿、隐私等场景 |
| 质检/运营 | 评估团队处理质量 | 查看评估报告、分析风险标记覆盖率、回溯决策记录 |
| AI 产品团队 | 优化分类和检索策略 | 运行离线评估、调整规则、对比指标变化 |

### 典型使用场景

| 场景 | 示例工单内容 | 系统行为 |
|------|-------------|----------|
| 售后退款 | "我买的手机有质量问题，申请退款" | 分类→refund，低风险，证据化草稿 |
| 投诉升级 | "再不处理我就找律师起诉" | 分类→complaint，legal_risk，强制人审 |
| 隐私/账号异常 | "我的账号在异地登录，个人信息可能泄露了" | 分类→account_issue，privacy+security风险，强制人审 |
| 发票/支付争议 | "付款成功但订单未确认，发票开不出来" | 分类→refund/other，policy_conflict，有风险信号时人审 |
| 证据不足转人工 | "退款。"（仅两字） | 分类→refund，insufficient_evidence，触发 fallback |

## 4. 产品目标与非目标

### Product Goals

- 提高工单分诊一致性——确保同一类型的工单获得统一的分类和处理路径
- 识别高风险工单——从自然语言中检测投诉、赔偿、法律、隐私等风险信号
- 帮助客服基于证据生成草稿——回复引用政策条款和相似案例，减少空口承诺
- 保留人工审核为最终决策层——高风险场景强制路由到人，不自动发送
- 建立可评估的数据闭环——101 条评测工单 + 离线评估流水线，可量化对比每次改动

### Non-goals（明确不做）

- ❌ 不做全自动客服——所有回复都是草稿，不自动发送
- ❌ 不替代人工客服——系统是辅助决策，不是替代人
- ❌ 不做生产系统——当前是本地 Demo / 作品集原型
- ❌ 不接真实企业数据——所有工单和知识记录都是合成/改编的
- ❌ 不声称真实线上效果——评测基于本地确定性数据，非线上 A/B test
- ❌ 不用 fake embedding 证明真实语义检索——当前嵌入仅验证管道机械正确性
- ❌ 不接真实 LLM——草稿用模板生成，避免 LLM 幻觉影响产品边界验证

## 5. 核心产品流程

### 流程图（文本版）

```
用户发送客服消息
     │
     ▼
[工单接收与标准化]
  → 文本规范化
  → 提取订单号、客户标识
     │
     ▼
[意图分类] — 8 类
  → 退款 / 退换货 / 账号 / 投诉 / 物流 / 技术 / 咨询 / 其他
     │
     ▼
[风险评估] — 8 种标记 + 3 级严重度
  → complaint / compensation / legal / privacy
  → account_security / policy_conflict / insufficient_evidence / low_confidence
  → severity: LOW → MEDIUM → HIGH
     │
     ▼
[分层知识检索]
  → 关键词全文搜索 + 向量相似度 + RRF 融合排序
  → 检索范围: FAQ(标准问答) + Policy(政策条款) + Case(历史案例)
     │
     ▼
[证据化草稿生成]
  → 基于检索到的 Policy/Case 生成回复草稿
  → 引用具体证据来源
  → unsupported claim guard: 不编造未检索到的政策
     │
     ▼
[人工审核]
  → 有风险标记 → must_human_review = true
  → 审核操作: 批准(Approve) / 编辑(Edit) / 升级(Escalate) / 拒绝(Reject)
  → 每次审核写入 ReviewDecision JSONL（审计追踪）
     │
     ▼
[离线评测与复盘]
  → CSV 模式: 加载样本预测 → 计算 7 项指标
  → Pipeline 模式: 运行完整管道 → 计算指标
  → 输出 JSON + Markdown 报告
```

### 每一步的产品价值

| 流程步骤 | 产品价值 | 对应系统能力 |
|----------|---------|-------------|
| 工单接收与标准化 | 让非结构化消息变成可处理的结构化数据 | intake/normalizer, entity_extractor |
| 意图分类 | 自动判断工单类型，减少人工分类成本 | classification/ 模块 |
| 风险评估 | 发现隐蔽风险信号，防止风险遗漏 | risk/assessor + rules |
| 分层知识检索 | 基于政策而非经验生成回复，提高一致性 | retrieval/ 完整链路 |
| 证据化草稿生成 | 客服不用从零写回复，提高效率和质量 | drafting/ provider + validator |
| unsupported claim guard | 防止系统编造不存在的政策 | drafting/citation_validator |
| 人工审核 | 高风险场景不自动处理，保留人的最终判断 | review/ console + store |
| 离线评测与复盘 | 每次改动可量化对比，不是"感觉变好了" | evaluation/ 全模块 |

## 6. 功能模块拆解

### 模块 1：工单输入与标准化

| 维度 | 说明 |
|------|------|
| 用户价值 | 直接粘贴客服消息即可处理，无需手动整理格式 |
| 业务价值 | 统一入口，保证后续流程的输入一致性 |
| 技术支撑 | RawTicket → NormalizedTicket 转换，实体提取（订单号、客户 ID） |
| 当前边界 | 仅支持单条文本输入，不支持批量导入和附件 |

### 模块 2：意图分类

| 维度 | 说明 |
|------|------|
| 用户价值 | 系统自动告诉客服"这是退款工单"还是"这是投诉工单" |
| 业务价值 | 减少人工分类时间，统一分诊标准 |
| 技术支撑 | 基于规则的 8 类分类器，不使用 LLM，确定性输出 |
| 当前边界 | 不支持自定义分类；单意图分类（复合意图归为主类） |

### 模块 3：风险识别与人审门控

| 维度 | 说明 |
|------|------|
| 用户价值 | 系统自动标记高风险工单，不需要客服逐条排查 |
| 业务价值 | 降低风险遗漏概率，保护平台免受法律/合规风险 |
| 技术支撑 | 8 种风险标记规则 + 严重度自动升级 + must_human_review 门控 |
| 当前边界 | 基于关键词规则，不是 ML 模型；部分边界案例可能漏报或误报 |

### 模块 4：FAQ / Policy / Case 分层知识库

| 维度 | 说明 |
|------|------|
| 用户价值 | 客服在一个系统里查标准问答、政策条款和历史案例，不用来回切换 |
| 业务价值 | 知识集中管理，回复基于最新政策而非个人经验 |
| 技术支撑 | 三种独立 doc_type，各带业务域标签，统一 chunk 存储 |
| 当前边界 | 95 条种子数据，不覆盖全部业务场景；非真实企业数据 |

### 模块 5：Hybrid Retrieval + RRF 证据检索

| 维度 | 说明 |
|------|------|
| 用户价值 | 系统自动找到最相关的政策条款和相似案例，减少客服查找时间 |
| 业务价值 | 回复一致性提升，证据可追溯 |
| 技术支撑 | 关键词全文搜索 + pgvector HNSW 向量检索 + RRF 融合排序 |
| 当前边界 | 向量部分使用 fake embedding，无语义检索质量；Phase 8 替换 |

### 模块 6：Evidence-based Draft

| 维度 | 说明 |
|------|------|
| 用户价值 | 客服看到草稿时知道"系统为什么这么写"——有证据引用 |
| 业务价值 | 减少空口承诺风险，新客服可参考草稿学习正确回复 |
| 技术支撑 | FakeDraftProvider 基于检索到的 EvidenceCandidate 生成模板化草稿 |
| 当前边界 | 模板生成，不调用 LLM，不是生产级回复质量 |

### 模块 7：Unsupported-claim Guard

| 维度 | 说明 |
|------|------|
| 用户价值 | 草稿不会编造政策——客服可以信任内容是有依据的 |
| 业务价值 | 降低法律风险，防止系统做出平台无法兑现的承诺 |
| 技术支撑 | CitationValidator 检查草稿中每个声明是否在检索结果中有依据 |
| 当前边界 | 基于规则扫描，复杂 claim 可能漏检 |

### 模块 8：Streamlit 人审控制台

| 维度 | 说明 |
|------|------|
| 用户价值 | 审核员在一个界面看到工单详情、风险评估、证据和草稿，直接操作 |
| 业务价值 | 完整的审核工作流 + 决策审计追踪（JSONL） |
| 技术支撑 | Streamlit 界面 + ReviewDecision 持久化 + JSONL 存储 |
| 当前边界 | MVP 级别，无登录、无多用户、无自动刷新 |

### 模块 9：Evaluation Pipeline

| 维度 | 说明 |
|------|------|
| 用户价值 | 产品团队可量化对比每次改动好不好 |
| 业务价值 | 不是"感觉变好了"而是数字说话——intent accuracy、risk flag recall 等 |
| 技术支撑 | CSV 模式 + Pipeline 模式，7 项指标，JSON + Markdown 报告 |
| 当前边界 | 101 条数据集不足以做统计显著分析；fake embedding 影响检索指标 |

### 模块 10：Demo Scenario Pack

| 维度 | 说明 |
|------|------|
| 用户价值 | 面试/作品集展示时，有 3 个完整场景可讲，不是零散截图 |
| 业务价值 | 向 stakeholders 展示产品能力时有一条清晰的叙事线 |
| 技术支撑 | 3 个场景文档 + 9 条示例工单 + 预期行为描述 + 限制说明 |
| 当前边界 | 场景示例是合成数据，不等于真实业务表现 |

## 7. 数据与评测体系说明

### 数据规模

| 数据集 | 数量 | 说明 |
|--------|:----:|------|
| Eval tickets（评测工单） | 101 | 合成中文单轮客服工单 |
| Golden expectations（黄金标注） | 101 | 产品规则驱动的预期行为标注 |
| Sample predictions（样本预测） | 101 | 全标记 no_auto_send=true |
| Knowledge records（知识记录） | 95 | FAQ=40 / Policy=30 / Case=25 |
| 覆盖的意图类型 | 8 | refund, return_exchange, account_issue 等 |
| 覆盖的风险标记类型 | 8 | complaint, compensation, legal, privacy 等 |
| 覆盖的严重度级别 | 3 | LOW=53, MEDIUM=33, HIGH=15 |
| 多意图工单 | 7 | refund+complaint, account+privacy 等 |
| 边界情况工单 | 5 | 单字符、超长文本、特殊字符等 |

### 数据来源策略

数据构造遵循"公开参考 → AI 辅助提取 → 人工改写 → 人工标注"四级流程：

```
公开/开源客服数据集（CSDS / Kaggle / 中文对话语料 / 公开政策页面）
    │ （作为场景和表达参考，不提交原始数据）
    ▼
AI 辅助字段提取（提取场景、风险信号、意图候选）
    │ （AI 是提取辅助，不是最终标注者）
    ▼
人工改写为合成工单（单轮中文客服工单，无真实个人信息）
    │ （人工改写，非原始数据复制）
    ▼
人工标注 Golden Expectations（基于产品规则，非 AI 生成）
    │
    ▼
进入评估流水线
```

关键原则：
- **不提交原始外部数据**到仓库
- **不把 AI 输出当最终标签**——所有 golden 标注经人工确认
- **不使用真实个人信息**——工单中均为占位数据

### 评测维度

| 维度 | 说明 | Pipeline 模式结果 |
|------|------|:----------------:|
| Intent accuracy | 意图分类是否正确 | 53.5% |
| Severity accuracy | 严重度级别是否正确 | 54.5% |
| Must-human-review accuracy | 是否需要人工审核判断是否正确 | 53.5% |
| Risk flag precision | 风险标记精确率 | 33.3% |
| Risk flag recall | 风险标记召回率 | 27.0% |
| Risk flag F1 | 风险标记 F1 | 29.8% |
| Evidence doc type recall | 证据文档类型召回率 | 43.2% |
| Fallback correctness | 降级处理是否正确 | 90.1% |
| No-auto-send compliance | 无自动发送合规率 | **100.0%** |

### 评测结果解读（产品视角）

**no-auto-send compliance = 100%** 是架构约束，不是回复质量指标。系统不连接任何发送通道，所以这条必然是 100%。它证明了"架构对齐"而非"效果很好"。

**Intent / Severity 指标（~53–54%）** 反映了当前 fake embedding 和基于规则的分类器的确定性行为。这些数字不高有两个原因：一是 fake embedding 不能做语义匹配，分类器只靠关键词规则；二是 pipeline mode 跑的是完整链路，意图分类的误差会向下游传递。这些指标说明"评测体系已建立"，但不能被包装成生产级效果指标。

**Risk flag 指标（F1 ~30%）** 类似——关键词规则对部分边界场景的覆盖不足。这是已知限制，不是异常。

**CSV 模式全部 100%** 说明当预测与 golden 严格对齐时，指标计算和报告输出是正确的。它验证了评测工具链本身无 bug。

### Demo 1：退款投诉

#### 作品集说明版

> 一条常见的售后退款消息，如果包含"投诉""赔偿""找律师"等语言，就从一个普通工单变成了高风险工单。TicketPilot 能同时检测三类风险标记——投诉风险、赔偿风险、法律风险——自动将严重度升级为 HIGH，强制路由到人工审核。草稿基于检索到的赔偿政策和相似案例生成，不凭空承诺赔偿金额。这个场景展示了系统的多意图识别能力、复合风险检测能力和高风险门控机制。

### 对比表

| 维度 | 普通 FAQ Bot | 普通 RAG 问答 | 全自动客服机器人 | 全渠道客服平台 | TicketPilot |
|------|:-----------:|:------------:|:--------------:|:------------:|:----------:|

| 证据引用 | ❌ | 有（不可控） | 部分 | ❌ | **显式引用 + 校验** |
| 人工审核 | ❌ | ❌ | 仅投诉升级 | 有 | **内置 + 必须** |
| 离线评测 | ❌ | ❌ | ❌ | ❌ | **7 项指标 + 两种模式** |

### 结论

TicketPilot 的优势不是"更强的聊天能力"，而是：

> **把 AI 放进客服工单处理流程里，并用风险标记、人工审核、证据引用和离线评测来约束它。**

它不是通用聊天机器人，不是文档 QA，不是全自动客服。它是一个**带边界、带流程、带评测的 AI 辅助决策原型**。这个定位让它特别适合作品集展示——因为它展示的不是"AI 能回答问题"，而是"AI 产品经理如何设计一个可控、可评测、有业务逻辑的 AI 工作流"。

## 13. 面试讲述

### 1 分钟版本

> "TicketPilot 是我做的一个中文客服工单处理系统原型。用户发一条售后消息，系统会先分类——这是退款、换货还是投诉——然后判断风险，有没有法律威胁、赔偿要求之类的。接着去知识库里查相关政策，基于证据生成一个草稿回复。但草稿不会自动发送，高风险工单强制人工审核。整个过程不调用 LLM，不接真实企业数据。我建了 101 条评测工单和离线评估流水线，做这件事的核心目的不是造一个客服机器人，而是证明我能设计一个带流程控制、风险判断、边界约束和评测体系的 AI 产品。"

### 3 分钟版本

> "TicketPilot 的出发点很明确：大多数 AI demo 都在展示'AI 能干什么'，我想展示的是'AI 产品的边界该怎么设计'。所以我没有做一个聊天机器人或全自动客服，而是做了一个工单分诊和辅助决策系统。
>
> 产品流程是这样的：用户提交客服消息→系统标准化→意图分类（8 类）→风险判断（8 种标记）→知识检索（FAQ/Policy/Case 三层）→生成带证据引用的草稿→unsupported claim guard 检查→人工审核决策。
>
> 几个关键设计决策：第一，不调 LLM，避免 LLM 幻觉干扰产品边界验证；第二，不自动发送，高风险工单强制人工审核；第三，用 fake embedding 验证管道正确性，不声称检索质量；第四，建立离线评测体系，intent accuracy ~53% 这个数字不够好看但诚实，因为 fake embedding 做不了语义匹配。
>
> 我做了数据策略——101 条工单不是从网上随便扒的，而是四级流程：公开数据做参考→AI 辅助提取→人工改写→人工标注。这样保证了数据的合规性和可追溯性。还做了 3 个 demo 场景文档、限制说明、作品集快照。Phase 8 会替换 fake embedding 做真实检索评估，但那是下一步了。"

### 追问回答

**你为什么做这个项目？**
> "初衷不是'我要做个客服 AI'，而是我想做一个能展示 AI 产品设计能力的项目——一个可控、可评测、有边界约束的 AI 工作流。客服工单场景刚好合适：流程清晰、风险明确、有业务判断逻辑。"

**这个项目解决了什么问题？**
> "它解决的是'AI 辅助决策怎么设计边界'的问题——不是 AI 能不能回答问题，而是 AI 怎么判断什么时候该回答、什么时候该转人工、回答的依据来自哪里。具体的业务痛点是工单分诊不一致、风险易遗漏、回复无追溯。"

**数据从哪里来？**
> "全部是合成数据。我们从 CSDS、Kaggle 等公开数据集取场景和表达参考，但不提交原始数据。每条工单都是人工改写的，不使用真实个人信息。Golden expectations 也是人工根据产品规则标注的。AI 只做字段提取辅助，不是最终标注者。"

**怎么评估？**
> "7 项指标：intent accuracy、severity accuracy、risk flag precision/recall/F1、evidence doc type recall、fallback correctness、must_human_review accuracy、no-auto-send compliance。CSV 模式和 Pipeline 模式双通道。no-auto-send=100% 是架构约束。其他指标受 fake embedding 限制，不能包装成生产级效果。"

**为什么不做全自动客服？**
> "因为全自动客服的技术和业务风险都很高：错误回复、法律承诺、无人工兜底。TicketPilot 的定位就是 AI 辅助+人审兜底。'不自动发送'是架构级的约束，不是配置开关。"

**你怎么保证 AI 不乱答？**
> "三层保护：第一，风险识别门控——有风险信号就强制人审；第二，unsupported claim guard——草稿中引用每个声明都必须在检索结果中有依据；第三，架构层——系统不连接任何发送通道。"

**当前效果有什么限制？**
> "主要限制是 fake embedding 不能做语义匹配，所以 intent/severity/risk flag 指标受限于关键词规则的覆盖。另外 101 条工单规模还不够做统计分析。草稿是模板生成的，不是生产级回复质量。这些都是已知限制，在文档里都写清楚了。Phase 10 的 doc-ID 评测已部分缓解了检索评估的语义需求，但 eval 数据的合成性和规模限制仍然存在。"

**下一步怎么优化？**
> "Phase 8–10 已完成。接下来可以做 query expansion audit（7 个 zero-hit 案例），或 fusion ranking 实验（32 个 partial-hit 案例），或接入 LLM 做证据化草稿生成。"

**这个项目体现了你的产品能力在哪里？**
> "第一，产品定位——我没有做一个全自动客服或聊天机器人，而是做了一个有边界控制的 AI 辅助决策系统。第二，需求拆解——把'客服 AI'拆成了分类、风险、检索、草稿、人审、评测六个独立可迭代的模块。第三，数据策略——设计了四级数据构造流程，确保合规性和可追溯性。第四，评测体系——7 维指标 + 双模式评测，不是暗箱操作。第五，场景包装——3 个跨风险类型的 demo 场景，每个都有明确的展示目标和边界说明。"

## 15. 项目边界提醒

### 不能这么说 ❌

| 不能说的话 | 原因 |
|-----------|------|
| "这是一个生产级客服系统" | 它是本地 Demo / 作品集，未经生产验证 |
| "已通过真实企业验证" | 所有数据是合成的，无真实企业数据 |
| "使用真实客户数据" | 使用的是合成数据，无真实 PII |
| "真实语义检索效果已验证" | 当前使用 fake embedding，Phase 8 才做真实评估；Phase 10 已用 dashscope text-embedding-v4 完成真实检索评估，Doc-ID Recall@10 = 91.9% |
| "系统自动发送客服回复" | 不连接任何发送通道，no-auto-send 是架构约束 |
| "可以替代人工客服" | 系统是辅助决策，不能替代人的判断 |
| "已通过线上 A/B 测试验证效果" | 评测是离线确定性的，非在线验证 |
| "达到行业 benchmark 水平" | 无行业 benchmark 对比，小规模合成数据不具可比性 |

### 应该这么说 ✅

| 正确的说法 | 原因 |
|-----------|------|
| "这是一个本地 Demo / 作品集原型" | 准确描述成熟度 |
| "使用合成/改编数据" | 说明数据来源策略 |
| "当前使用 fake embedding 验证管道机械正确性" | 说明嵌入的定位 |
| "草稿仅用于人工审核参考，不自动发送" | 说明 no-auto-send 是架构约束 |
| "AI 辅助客服决策，不替代人工" | 说明人机协作定位 |
| "离线评测验证产品流程正确性，非线上效果" | 说明评测定位 |
| "当前指标受限于 fake embedding，不代表真实检索性能" | 诚实说明限制；Phase 10 已用真实 embedding 完成 doc-ID 检索评估 |
