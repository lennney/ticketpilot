## 1. 项目概览

TicketPilot 是一个面向中文电商售后场景的工单分诊与证据化回复 Copilot 原型。用户提交一条中文客服消息，系统自动完成意图分类、风险识别、分层知识检索和证据化草稿生成，最后由人工审核完成决策闭环。

它不是聊天机器人，也不是全自动客服。它的核心定位是：把 AI 放进客服工单处理流程，用风险标记、证据引用、人工审核和离线评测来约束它，而不是让它代替人做判断。

当前项目已完成 Phase 7–10，Phase 11（证据约束 LLM 草稿生成）进行中。全流程支持默认模板生成和可切换的 LLM 证据约束草稿；嵌入支持默认 fake 和 opt-in 真实 provider（DashScope text-embedding-v4）；数据全部为合成数据。

## 2. 问题背景

- 工单类型混杂：一条退款消息可能同时包含投诉情绪、赔偿要求甚至法律威胁
- 分诊标准不统一：同样的退款问题，不同客服可能给出不同的处理路径
- 风险容易被遗漏：一句"不然我找律师"写在退款请求里，如果没识别出法律风险，后果严重
- 回复质量依赖个人经验：缺乏统一标准和证据追溯

## 3. 核心用户流程

```
用户提交中文工单
  → 工单标准化与实体提取
  → 意图分类（8 类）
  → 风险判断（8 种标记 + 3 级严重度）
  → FAQ / Policy / Case 分层知识检索（106 条记录）
  → 证据化回复草稿生成（模板 / LLM 约束 + ClaimGuard）
  → 人工审核（Approve / Edit / Escalate / Reject）
  → 离线评测与复盘
```

## 4. 迭代历程（核心差异化）

| Phase | 动作 | 关键发现 | 产品判断 |
|-------|------|---------|---------|
| 7 | 建数据基底（10→101 条工单） | 评测体系建成 | — |
| 8 | 换真实嵌入（DashScope text-embedding-v4） | Fake/Real 的 41 个错例完全一致 | 瓶颈在知识覆盖，不在模型 |
| 9 | 定向补 11 条知识 | `load_dotenv()` 未调用，评测静默回退 | Provider Identity Gate 必须建立 |
| 10 | 细粒度评测（doc-ID） | Doc-ID Recall@10=91.9%，78% wrong cases 是粒度问题 | 评估方法不对会误判系统 |
| 11 | LLM 草稿生成（进行中） | — | 检索质量达标，转向生成优化 |

## 5. 数据与评测

| 指标 | 数据 |
|------|------|
| Eval tickets | 101 条合成工单 |
| Knowledge records | 106 条（FAQ=41, Policy=34, Case=31） |

| 评测维度 | 7 项指标 + 细粒度 doc-ID 指标 |
| Doc-ID Recall@10 | **91.9%**（Phase 10 细粒度评测） |
| Fake vs Real Top-1 | 31.7% → **42.6%**（+10.9%） |

## 6. 差异化定位

| 维度 | 普通 RAG | TicketPilot |
|------|---------|-------------|
| 流程 | 问答 | 分诊 + 风控 + 检索 + 审核 |
| 风险控制 | ❌ | 8 种标记 + 强制人审 |
| 评测 | ❌ | 多维指标 + 迭代验证 |
| 迭代轨迹 | ❌ | 5 阶段，每步有产品判断 |

## 7. 关键限制

- 本地 Demo / 作品集原型
- 全部合成数据
- 默认 fake embedding（真实嵌入 opt-in）
- 草稿为模板生成 / FakeLLM，不调真实 LLM
- 不自动发送（no-auto-send 架构约束）
