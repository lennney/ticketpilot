## 1. 项目概览

TicketPilot 是一个面向中文电商售后场景的工单分诊与证据化回复 Copilot 原型。用户提交一条中文客服消息，系统自动完成意图分类、风险识别、分层知识检索和证据化草稿生成，最后由人工审核完成决策闭环。

它不是聊天机器人，也不是全自动客服。它的核心定位是：把 AI 放进客服工单处理流程，用风险标记、证据引用、人工审核和离线评测来约束它，而不是让它代替人做判断。

当前项目已完成 Phase 8（Real Retrieval Upgrade），在 Phase 7 数据集下完成 fake vs real embedding retrieval comparison。全流程不使用真实 LLM，不连接发送通道，所有数据为合成数据。详情见 [Phase 8 Snapshot](phase8_real_retrieval_snapshot.md)。

## 2. 问题背景

中文电商客服工单处理面临几个结构性问题：

- **工单类型混杂**：一条退款消息里可能同时包含投诉情绪、赔偿要求甚至法律威胁。一线客服需要在短时间内判断主诉求、识别隐藏风险、查找相关政策、撰写回复。
- **分诊标准不统一**：同样的退款问题，不同客服可能给出不同的处理路径，影响用户体验和团队效率。
- **风险容易被遗漏**：一句"不然我找律师"写在退款请求里，如果没识别出法律风险，可能从普通投诉升级到法律纠纷。
- **回复质量依赖个人经验**：新客服培训周期长，老客服的回复风格和依据各不相同，难以追溯和复盘。
- **现有方案各有盲区**：纯 FAQ 机器人无法处理复合意图；通用 RAG 问答缺少风险判断和流程控制；全自动客服在高风险场景下错误承诺的风险过高；大部分 AI demo 没有评测体系和人审闭环。

所以 TicketPilot 选择了一条不同的路——不做全自动客服，而是做一个"工单分诊 + 风险判断 + 证据化草稿 + 人工审核"的 AI 辅助决策原型，并用离线评测证明它能被衡量和迭代。

## 5. 核心用户流程

```
用户提交中文工单
  → 工单标准化与实体提取（订单号、客户标识）
  → 意图分类（8 类）
  → 风险判断（8 种标记 + 3 级严重度）
  → FAQ / Policy / Case 分层知识检索
  → 证据候选输出
  → 证据化回复草稿生成
  → Unsupported-claim guard 检查
  → 人工审核（Approve / Edit / Escalate / Reject）
  → 离线评测与复盘
```

| 流程步骤 | 产品价值 | 系统能力 |
|---|---|---|
| 工单标准化 | 非结构化消息变为可处理的结构化数据 | 文本规范化 + 实体提取 |
| 意图分类 | 自动判断工单类型，减少人工分类成本 | 8 类确定性规则分类器 |
| 风险判断 | 发现隐蔽风险信号，防止遗漏升级 | 8 种风险标记 + 严重度自动升级 |
| 分层知识检索 | 基于政策而非个人经验生成回复 | 关键词 FTS + 向量 HNSW + RRF 融合 |
| 证据化草稿 | 客服不用从零写回复 | 模板生成 + 证据引用 |
| Unsupported-claim guard | 防止系统编造不存在的政策 | 声明与检索结果的交叉校验 |
| 人工审核 | 高风险场景不自动处理，保留人的判断 | Streamlit 控制台 + JSONL 审计追踪 |
| 离线评测 | 每次改动可量化对比 | CSV 模式 + Pipeline 模式 + 7 维指标 |

## 6. 核心功能模块

### 6.1 工单输入与标准化

用户粘贴客服消息即可处理，无需手动整理格式。系统提取订单号、客户 ID 等关键字段，确保后续流程的输入一致性。当前仅支持单条文本输入，不支持批量导入和附件。

### 6.2 意图分类

系统自动判断工单属于退款、退换货、账号问题、投诉、物流、技术、咨询还是其他——共 8 类确定性分类，不使用 LLM，输出可复现。当前不支持自定义分类或复合意图拆分。

### 6.3 风险识别与人审门控

从自然语言中检测 8 种风险信号：投诉、赔偿、法律、账号安全、隐私、政策冲突、证据不足、低置信度。系统自动评定 LOW / MEDIUM / HIGH 三级严重度，存在任意风险标记即触发人工审核。基于关键词规则，边界案例可能漏报或误报。

### 6.4 FAQ / Policy / Case 分层知识库

知识库按三种文档类型组织：FAQ 存放标准问答、Policy 存放政策条款、Case 存放历史案例。每种类型携带业务域标签，客服在一个系统内即可查询三类信息，无需切换工具。当前 95 条种子数据，覆盖评测场景但不代表真实业务分布。

### 6.5 Hybrid Retrieval + RRF 证据检索

结合关键词全文搜索和向量相似度检索，通过 RRF 算法融合排序，从三层知识库中找到最相关的证据候选。当前向量部分使用 fake embedding（确定性哈希），仅验证管道机械正确性，无语义检索质量。

### 6.6 证据化回复草稿

基于检索到的 Policy 和 Case 生成模板化回复草稿，每项声明附带证据来源引用。客服看到草稿时知道"系统为什么这么写"。当前是模板生成，不调用 LLM，不是生产级回复质量。

### 6.7 Unsupported-claim Guard

自动检查草稿中的每个声明是否在检索结果中有依据，防止系统编造不存在的政策条款。这是降低法律风险的关键防线。当前基于规则扫描，复杂声明可能漏检。

### 6.8 Streamlit 人审控制台

审核员在一个界面看到完整的工单详情、风险评估、证据列表和回复草稿，可执行 Approve / Edit / Escalate / Reject 四种操作。每次决策写入 JSONL 文件形成审计追踪。MVP 级别，无登录、无多用户、无自动刷新。

### 6.9 Evaluation Pipeline

CSV 模式加载样本预测文件快速计算指标；Pipeline 模式完整运行每条工单的整个链路。7 项指标覆盖意图、风险、证据、人审等关键决策节点。101 条数据集不足以做统计显著分析。

### 6.10 Demo Scenario Pack

3 个完整演示场景（退款投诉、隐私/账号异常、发票/支付争议），每个场景包含 3 条示例工单、预期流程、风险行为和边界说明，支撑作品集展示和面试讲述。

## 7. 数据与评测体系

### 数据规模

| 数据集 | 数量 |
|---|---|
| Eval tickets（评测工单） | 101 |
| Golden expectations（黄金标注） | 101 |

| Policy | 30 |
| Case | 25 |

### 数据来源策略

数据构造遵循"公开参考 → AI 辅助提取 → 人工改写 → 人工标注"四级流程：

- 公开/开源客服相关数据集（CSDS、Kaggle 等）只作为场景、表达和结构参考，原始数据不提交到仓库；
- 最终评测工单是人工改写的合成中文单轮客服工单，不使用真实个人信息；
- Golden expectations 是人工按产品规则确认的，AI 只辅助字段拆解，不是最终标注者；
- 知识记录为常见电商客服场景的合成/改编内容，不来源于任何真实企业的知识库或政策文档。

### 评测维度

| 维度 | Pipeline 模式结果 | 说明 |
|---|---|---|
| Intent accuracy | 53.5% | 意图分类是否准确 |
| Severity accuracy | 54.5% | 严重度判定是否准确 |
| Must-human-review accuracy | 53.5% | 人审路由判断是否准确 |
| Risk flag precision | 33.3% | 风险标记精确率 |
| Risk flag recall | 27.0% | 风险标记召回率 |
| Risk flag F1 | 29.8% | 风险标记 F1 |
| Evidence doc type recall | 43.2% | 证据文档类型召回率 |
| Fallback correctness | 90.1% | 降级处理是否正确 |
| No-auto-send compliance | 100.0% | 无自动发送合规率 |

### 边界说明

- **No-auto-send compliance = 100%** 是架构约束，不是回复质量指标。系统不连接任何发送通道。
- **当前 fake embedding** 只验证管道机械正确性，余弦相似度无语义含义。向量检索不能证明语义理解能力。
- **当前 intent / severity 指标**（~53–54%）反映了 fake embedding 和基于规则的组件的确定性行为。这些指标说明评测体系已建立，但不能被包装成生产级效果。
- **Phase 8** 已完成 DashScope text-embedding-v4 接入与对比评测。Top-1 hit rate 从 31.7%（fake 384-d）提升至 42.6%（real 1024-d），MRR 从 0.4114 提升至 0.4913。错例分析显示 41 个 wrong cases 均为 missing_doc_type，当前瓶颈是知识库覆盖而非 embedding 质量。详见 [Phase 8 Snapshot](phase8_real_retrieval_snapshot.md)。

## 9. 三个强 Demo 场景

### Demo 1：退款投诉

**示例工单：** "我申请退款三天了还没到账，订单号是DEMO-REFUND-001。你们再不处理我就投诉，还要赔偿。"

**关键风险：** complaint_risk（投诉风险）+ compensation_risk（赔偿风险）+ legal_risk（法律风险）

**系统处理：** 分类为 refund / complaint，识别三类风险标记，严重度自动升级为 HIGH，强制路由到人工审核。草稿基于赔偿处理规则和律师函处理案例生成，不承诺具体赔偿金额。

**展示能力：** 多意图检测、复合风险叠加与严重度自动升级、高风险强制人工审核、证据驱动的回复草稿、unsupported-claim guard。

**截图建议：** 工单输入 → 风险评估（3 个标记 + HIGH + must_human_review）→ Evidence candidates（Policy + Case）→ Draft reply（高风险警告）→ 审核控制台。

---

### Demo 2：隐私/账号异常

**示例工单：** "我的账号出现异地登录，而且手机号和实名信息好像被别人看到了，请马上处理。"

**关键风险：** account_security_risk（账号安全风险）+ privacy_risk（隐私泄露风险）

**系统处理：** 分类为 account_issue，识别双风险标记，判定 HIGH severity，强制人工审核。草稿基于隐私政策和账号安全流程生成，提供账号保护指引但不代用户执行敏感操作。

**展示能力：** 隐私安全类风险检测、安全场景的 HIGH 判定、草稿的边界意识（不承诺操作、不要求明文密码）。

**截图建议：** 工单输入 → 风险评估（account_security + privacy + HIGH）→ Evidence（Policy + Case）→ Draft reply（不承诺操作的边界）→ Evaluation report 的 risk flag 指标。

---

### Demo 3：发票/支付争议

**示例工单：** "我付款成功了但订单显示未支付，还重复扣了一笔，发票也开不出来。订单号：DEMO-PAY-001。"

**关键风险：** policy_conflict（政策冲突），MEDIUM severity；无风险标记时（如纯发票咨询）不必人工审核。

**系统处理：** 分类为 refund / other，检测 policy_conflict，MEDIUM severity，触发人工审核。纯发票咨询（如 C2）无风险标记时无需人审。草稿基于发票规则和支付政策生成，不承诺退款。

**展示能力：** 条件性人工审核（有风险标记 → 人审，无风险标记 → 自动处理）、领域特定检索、政策对齐、多意图区分。

**截图建议：** 工单输入 → 风险评估 → Evidence（Policy + Case）→ Draft reply → 对比有/无风险标记时的不同人审行为。

## 10. 产品差异化

| 维度 | FAQ Bot | 普通 RAG 问答 | 全自动客服机器人 | TicketPilot |
|---|---|---|---|---|
| 工单分诊 | 低（仅匹配问题） | 中（检索相关文档） | 中 | 高（分类 + 风险 + 人审流程） |
| 风险识别 | 弱 | 弱 | 不稳定 | 明确规则（8 种标记） |
| 证据化回复 | 弱（固定答案） | 中（可能幻觉） | 不稳定 | 强制证据候选 + 引用校验 |
| 人工审核 | 通常无 | 通常无 | 仅投诉升级 | 核心流程门控 |
| 离线评测 | 弱 | 通常缺失 | 不透明 | 7 维指标 + 两种模式 |
| 适合高风险售后 | 低 | 中 | 风险高 | 更可控 |

**结论：** TicketPilot 的差异化不是"更会聊天"，而是把 AI 放进客服工单处理流程，并用风险标记、人工审核、证据引用和离线评测来约束它。它不是通用聊天机器人，不是文档 QA，不是全自动客服——它是一个带边界、带流程、带评测的 AI 辅助决策原型。

## 11. 我的贡献

- **产品定位**：从"做一个客服 AI"转向"做一个可控、可评测、有边界约束的 AI 辅助决策系统"。明确不做全自动客服、不接 LLM、不自动发送。
- **场景与需求拆解**：把"客服工单处理"拆解为工单输入、意图分类、风险识别、知识检索、草稿生成、人工审核、离线评测七个独立模块，每个模块有明确的产品价值、技术支撑和当前边界。

- **知识库结构设计**：设计了 FAQ / Policy / Case 三层知识结构，按业务域组织，支撑意图分类和证据检索的联合查询。
- **评测体系搭建**：定义了 7 维评测指标，设计 CSV 和 Pipeline 双模式评测，建立 101 条 golden expectations 标注基准。指标不高但诚实——fake embedding 的限制在文档中明确说明。
- **Demo 与边界包装**：设计了 3 个跨风险类型的演示场景，编写了完整的限制说明文档，确保项目不被过度包装。

## 12. 项目限制

- **本地 Demo / 作品集原型**：不是生产级客服系统，未经过安全审计、负载测试或合规审查。
- **合成数据**：所有工单和知识记录为人工编写或改编的合成数据，不使用真实企业客户数据。
- **Fake embedding**：默认使用 FakeEmbeddingProvider（确定性 SHA-256 哈希），余弦相似度无语义含义，仅验证管道机械正确性。
- **无真实 LLM**：草稿生成使用确定性模板（FakeDraftProvider），不调用任何 LLM API。
- **Draft-only / no auto-send**：系统不连接任何发送通道。所有输出为草稿，人工审核仅写入本地 JSONL。
- **离线评测**：评估基于合成数据和确定性规则，非线上 A/B test 或真实业务 benchmark。101 条数据集不足以支持统计显著分析。
- **当前指标**：Pipeline mode 的 intent accuracy（53.5%）、severity accuracy（54.5%）、risk flag F1（29.8%）反映了 fake embedding 和规则组件的确定性行为，不能被包装成生产效果。No-auto-send compliance = 100% 是架构约束。
- **UI 级别**：Streamlit 人审控制台为 MVP 级别，无身份验证、无多用户支持。
